## 对本项目最有价值的观察

**"国内报表对字段级配色要求细致（"字号 16、微软雅黑、加粗、#5DECEC"）是 BI 工程师习惯"**——Smartbi 文档显示国内 BI 用户对每个字段的字体、颜色、加粗、字号都有强期望。**本项目应在 `templates/` 中提供"样式 token 字典"**，把常量的字体 / 字号 / 色值集中定义（如 `FONT_TITLE = "16px 微软雅黑 bold #FFFFFF"`），便于月度迭代时 HR 部门反映"某个字段字号调大"时一处改全局生效。

**"Tab 多页签是中国式报表特有模式"**——Smartbi 用 TAB 组件切换"电子账单 / 纸质账单"，同屏展示一组对比数据。本项目 M-2 (年龄分布模块) 可考虑用 Tab 切换"全行 vs 部门"两套视图，复用一个图表实现。

**"国内 BI 强调`机构码 / 权限 / 多层级`数据模型"**——本项目虽不做权限，但 M-6 (客户经理模块) 应**预先在 Excel sheet 中定义"按支行分组的字段"**，未来扩展权限留口子。

## 对本项目不适合的点

**依赖 Smartbi 服务端 XML 配置（资源 20MB+）**——Smartbi demo 是 .xml 资源 + .rar 数据库样本，完全不能单文件分发。**仅借鉴"中国式报表"对字体的细致要求和 Tab 组件思路**，不借鉴服务端架构。

**强调"配置文档详尽"导致上手门槛高**——本项目目标是 HR 1 行命令 `python build.py` 跑通；Smartbi 的"配置文档逐字段"模式反而暴露了 BI 工具对最终用户的"上手成本"。本项目要保持**自动生成 + 0 配置**的极简。
