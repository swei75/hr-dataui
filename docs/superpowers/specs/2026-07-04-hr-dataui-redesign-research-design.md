## 0. 背景与动机

### 0.1 问题陈述

hr-dataui v1.3 已交付，但用户反馈"界面丑、字体样式不专业"，并且和真实 .docx HR 月报（`docs/范例：人力资源管理数据驾驶舱（2026年5月）V2.docx`）的格式/详略不匹配。

### 0.2 已发现的两个差距

**差距 A — 信息架构**：
- 真实 .docx 写法：段落叙述 + 关键数字 + 三级排名对比 + 多列对比表
- 当前 dashboard：纯数据 + 图表，缺叙述，缺小节，缺对比维度

**差距 B — 视觉风格**：
- 当前：单一 indigo + zinc 灰，14px 字号，18px 内边距
- 用户评价："不专业"

### 0.3 关键约束（**不可动**）

即使研究发现某些做法更优，以下约束**不允许修改**：

| 不可动项 | 理由 |
|---|---|
| C-1 单文件 HTML 输出 | 邮件 / U 盘 / 内网分发的核心场景 |
| C-2 0 外部依赖（无 CDN） | 接收方无技术背景 |
| C-3 完全离线 / file:// 可用 | 避开 CORS；任何 fetch 都是负担 |
| C-4 多端响应式 | 领导可能在手机看 |
| C-5 体积目标 ≤ 50KB | 邮件附件友好（当前 69KB，可接受） |
| ADR-001 单 Excel 数据源 | HR 月报场景已固化 |

> 任何 demo 即使更好，只要违反 C-1~C-5 就不构成 ADR 修订证据。

## 2. 研究方法

### 2.1 三个深研究 — 来源策略

**核心难点**：国内银行 HR 月报基本都是内部材料，公开 HTML demo 极少。

**策略（按可行性排序）**：

| 优先级 | 来源 | 备选 |
|---|---|---|
| 🟢 1st | **GitHub 搜索中文关键词**（"银行 HR 仪表盘"、"dashboard 报告 html"、"hr 月报"） | 找 fork 关系；爬同主题项目 |
| 🟡 2nd | **公开 BI 工具的"银行/HR"案例**（Tableau Public / 帆软 / Smartbi） | 截图 + 导出 HTML |
| 🔴 3rd | **上市公司年报嵌入的 HR 数据图** | 仅概念参考，无 HTML |

**3 个深研究目标画像**：

| # | 目标 | 对应 .docx 特征 |
|---|---|---|
| 深 #1 | 中文 HR 月报 demo（最像 .docx） | 叙述 + 三级排名 + 多列对比表 |
| 深 #2 | 银行/金融 dashboard（行业基准） | 配色稳重 + 数据密度高 |
| 深 #3 | BI 工具"管理驾驶舱"（领导视角） | KPI 突出 + 异常告警 |

**Fallback 顺序**（某优先级找不到）：
- GitHub 没 → Tableau Public 找中文 HR 主题
- Tableau 没 → 帆软/Smartbi 公开 demo
- 都没 → 用 .docx 本身作概念参考（无需 HTML）

### 2.2 八个广扫描 — 来源清单

| # | 来源 | 类型 |
|---|---|---|
| 1 | Tableau Public - HR Analytics | 国际 BI |
| 2 | Tableau Public - Finance Dashboard | 国际 BI |
| 3 | PowerBI Showcase - HR | 国际 BI |
| 4 | 帆软 FineBI 官方 demo - HR 场景 | 国内 BI |
| 5 | 帆软 FineBI 官方 demo - 银行/金融场景 | 国内 BI |
| 6 | Smartbi 公开 demo | 国内 BI |
| 7 | Apache Superset 官方 example | 开源 BI |
| 8 | Metabase 官方 example | 开源 BI |

> **至少 4 个国内 BI**（FineBI + Smartbi）保证中文语境样本。

### 2.3 深研究的 15 个观察维度（每份 README 必填）

1. **结构层级**：模块 / 章节 / 小节 / 数据点——几层？
2. **叙述 vs 数据比例**：文字段落 vs 图表/数字——几比几？
3. **图表类型清单**：用了哪些 viz？为什么？
4. **KPI 展示**：单数字 / 多数字 / 趋势？
5. **排名展示**：Top N 用什么形式？（徽章 / 柱图 / 表格？）
6. **对比手法**：完成率 vs 完成量 vs 进度——怎么同时呈现？
7. **配色方案**：主色 / 辅色 / 中性色（取色记录）
8. **字体系统**：标题 / 副标题 / 正文 / 数字——字号与字重
9. **间距节奏**：模块 / 卡片 / 数据点——几 px？
10. **强调手段**：颜色 / 粗体 / 背景 / 图标——如何突出重点？
11. **表格用法**：什么场景用表格？表格如何与图表配合？
12. **模块标题样式**：编号、装饰、位置
13. **留白比例**：信息密度 vs 呼吸感
14. **银行/政务感**：配色 + 字体 + 排版的整体气质
15. **整体印象**：3 个形容词

### 2.4 广扫描对比矩阵（9 列）

| 列 | 说明 |
|---|---|
| demo 名称 | 链接 |
| 模块数 | 数量 |
| 章节深度 | 1/2/3 层 |
| 叙述占比 | 0-100% 估算 |
| 图表类型 | 列表 |
| 主色 | hex |
| 信息密度 | 稀疏/中等/密集 |
| 银行感 | 0-5 评分 |
| 整体印象 | 3 个形容词 |

## 3. 决策映射框架

### 3.1 三个判定类型

每个 ADR 给出**唯一**判定：

| 判定 | 含义 | 触发条件 |
|---|---|---|
| **KEEP** | 决策继续有效 | <2 个 demo 反驳 + 用户认可 |
| **REVISE** | 决策方向对，修订具体点 | ≥3 个 demo 展示更好的局部做法 + 用户同意方向 |
| **DEPRECATE** | 决策被取代，新增 ADR | ≥5 个 demo 展示根本不同做法，**或** 1 个深 demo 完美对应 |

### 3.2 证据等级

| 来源 | 强度 | 用途 |
|---|---|---|
| 深研 #1（中文 HR 月报） | 🟢🟢🟢 最强 | 直接对标 .docx |
| 深研 #2（银行/金融 dashboard） | 🟢🟢 强 | 行业基准 |
| 深研 #3（管理驾驶舱） | 🟢🟢 强 | 领导视角标杆 |
| 广扫描（BI 工具） | 🟡 中 | 横向参考，不单独构成证据 |

> **原则**：单个广扫描不能推翻 ADR；3 个深研究任 1 个反驳就足以触发 REVISE 讨论。

### 3.3 决策流程

```
研究完成（3 深 + 8 广 + 矩阵）
    │
    ▼
Claude 填充决策映射表初稿
    │
    ▼
逐条 ADR 跟用户过：
   - 呈现：判定 + 证据摘要
   - 用户：同意 / 反驳 / 改判定
    │
    ▼
最终版写入 ARCHITECTURE.md
（新增"研究驱动的修订建议"章节，旧 ADR 历史保留）
```

### 3.4 已知不动项（即使研究显示不同）

| 不可动 ADR | 理由 |
|---|---|
| ADR-001 单 Excel 数据源 | HR 月报场景已固化 |
| ADR-005 Python f-string | 零依赖，符合 C-2 |
| ADR-010 单文件部署 | 核心约束 C-1 重申 |

> 这 3 个即使被 demo 反驳，仍标 KEEP（不可动），并在判定栏注明原因。

- [ ] § 来源（仓库链接 / 公开 URL / 截图来源）

### 4.3 广扫描矩阵必填项

- [ ] 8 行 × 9 列全部填实（不留 TBD）
- [ ] 每个 demo ≥ 1 句具体观察（不是"看起来专业"，而是"用 #1E40AF 主色 + 12px 圆角"）
- [ ] 矩阵下方有 § 汇总观察（按列横向对比发现）

### 4.4 决策映射必填项

- [ ] 10 个 ADR 全部给出 KEEP/REVISE/DEPRECATE 判定
- [ ] 每个判定引用至少 1 个具体 demo 作为证据
- [ ] REVISE/DEPRECATE 必须给出**具体修订方向**（不是"考虑加 X"，而是"ADR-002 增加第 8 列 `叙述`"）
- [ ] 不可动项的 ADR 即使研究不反驳也要明确标注"KEEP（不可动）"
